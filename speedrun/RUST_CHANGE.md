# Rust Scheduler Change — Points-at-Stake Review Order

## What was changed

A new review card ordering mode called **Points-at-Stake** was added to Anki's Rust scheduler. When this mode is active, due cards are surfaced in descending order of:

```
points_at_stake = section_weight × (1 − FSRS_retrievability)
```

A card with high section weight (important MCAT topic) and low retrievability (likely forgotten) floats to the top of the review queue. A card with low weight or high retrievability sinks toward the bottom.

**Section weights** are tunable constants defined in `rslib/src/storage/card/mod.rs`:

| Subdeck | Weight |
|---------|--------|
| Behavioral | 1.0 |
| Biology | 0.5 |
| Biochemistry | 0.5 |
| General-Chemistry | 0.33 |
| Organic-Chemistry | 0.33 |
| Physics-and-Math | 0.33 |
| Essential-Equations | 0.15 |

CARS has no flashcard deck and receives an implicit weight of 0.0.

## Why it was changed

Standard Anki review orders sort by due date, relative overdueness, or retrievability — none of which account for how much a topic matters on the target exam. For MCAT preparation, a card in Behavioral (worth ~25% of one section) is more valuable to review than an equally forgotten card in Essential-Equations. Surfacing the highest-value weak cards first maximises exam-score improvement per unit of study time.

## How it fits into Anki's existing architecture

Anki's scheduler builds the review queue in Rust (`rslib/src/scheduler/queue/builder/`) using a SQL query against the `cards` table. The ORDER BY clause of that query is generated by `review_order_sql()`, which dispatches on the deck config's `ReviewCardOrder` enum value to produce a `ReviewOrderSubclause`.

The Points-at-Stake implementation follows this exact pattern — it adds one new enum variant and one new `Display` implementation that emits a SQL subselect:

```sql
(select case
    when instr(char(31)||d.name||char(31), char(31)||'Behavioral'||char(31)) > 0 then 1.0
    when instr(char(31)||d.name||char(31), char(31)||'Biology'||char(31))    > 0 then 0.5
    ...
    else 0.0 end
 from decks d where d.id = (case when odid != 0 then odid else did end))
 * (1 - extract_fsrs_retrievability(data, ..., today, next_day_at, now))
 desc
```

The deck name is matched component-wise using the `\x1f` (char(31)) separator that Anki already uses internally to store full deck paths. The `odid != 0` guard ensures cards inside filtered decks are looked up by their home deck, preserving correct topic weights when cards are temporarily moved for cramming.

`extract_fsrs_retrievability` is Anki's existing Rust SQL function — the Points-at-Stake mode reuses it rather than reimplementing the FSRS formula. The FSRS scheduling itself (stability updates, interval calculation, ease ratings) is completely untouched. Points-at-Stake only affects which already-due cards are shown first within a given study session; it does not change when cards become due or how intervals grow.
